# 决策矩阵：LoongArch64 移植决策指南

> **Agent 阅读指引**
> 
> 当你不确定如何处理移植过程中的某个情况时，使用本文档的决策流程。
> 
> 本文档的目标是：在保持移植效率的同时，防止源码污染和误移植。

## 1. 移植阶段决策流程图

```
开始移植任务
  │
  ▼
┌─────────────────────────────┐
│ Git仓库是否已初始化？          │──否──▶ 执行: git init && git add -A
│ 且已有初始提交？               │        && git commit -m "Initial import"
└─────────────────────────────┘        然后继续
  │是
  ▼
┌─────────────────────────────┐
│ 当前处于移植的哪个阶段？       │
└─────────────────────────────┘
  │
  ├─ 阶段0/1（首次接触）──────────▶ 执行: 信息收集
  │                                  查找README/docs/build.md/ebuild/PKGBUILD
  │                                  识别构建系统
  │
  ├─ 阶段2（X86-64参考构建）──────▶ 执行: X86-64构建
  │                                  按文档步骤构建
  │                                  记录完整步骤和测试输出
  │
  ├─ 阶段3（LoongArch64移植）─────▶ 执行: 架构确认 → 清理build → 构建 → 修复
  │                                  见下方"阶段3子流程"
  │
  └─ 阶段4（移植交付）────────────▶ 执行: 生成补丁 + 撰写报告

  │

│ 是否为 loongarch64？          │──否──▶ 询问用户是否切换架构

  │
  ▼

  │
  ▼

  重新构建

  ▼

【阶段4子流程：移植交付】

构建和测试均通过
  │
  ▼

  │
  ▼

  ▼
```

## 2. 快速决策表

### 2.1 按移植阶段

| 阶段 | 关键决策 | 风险等级 | Agent行动 |
|------|----------|----------|-----------|
| 阶段0：源码保护 | 是否已Git初始化？ | 🔴 L0 | 未初始化则**立即停止并执行** |
| 阶段1：信息收集 | 找不到构建文档怎么办？ | 🟡 L2 | 按优先级扫描文件，仍找不到则**询问用户** |
| 阶段2：X86-64构建 | X86-64构建失败怎么办？ | 🟡 L2 | 修复通用问题，**不急于移植** |
| 阶段3：LA64构建 | 架构检测失败（unknown arch） | 🟡 L2 | 按 la64-porting-guide.md 第1节处理 |
| 阶段3：LA64构建 | 内联汇编编译错误 | 🟠 L1 | **询问用户**修改方案，参考 la64-porting-guide.md 第3节 |
| 阶段3：LA64构建 | 链接错误（符号未定义） | 🟡 L2 | 检查是否缺少LA64库或需条件编译 |
| 阶段3：LA64构建 | 类型/对齐警告 | 🟡 L2 | 分析后修复，使用stdint类型 |
| 阶段3：LA64构建 | SIMD头文件找不到 | 🟡 L2 | 按 la64-porting-guide.md 第2节使用SIMDE替换 |
| 阶段3：LA64构建 | `__rdtsc` / `_mm_pause` 未定义 | 🟡 L2 | 按 la64-porting-guide.md 第3节替换 |
| 阶段3：LA64构建 | CMake SYSTEM_PROCESSOR判断缺失 | 🟡 L2 | 按 la64-porting-guide.md 第4.2节添加 |
| 阶段3：LA64构建 | spec ExclusiveArch 不含 loongarch64 | 🟡 L2 | 按 la64-porting-guide.md 第9.4.1节添加架构声明 |
| 阶段3：LA64构建 | rpmbuild 构建依赖缺失 | 🟡 L2 | 检查 BuildRequires 在 LA64 上是否可用，按第9章处理 |
| 阶段1：信息收集 | 项目仅有 spec 文件无 README | 🟡 L2 | 从 spec 提取 BuildRequires/%build/%check，按 domain-knowledge §7.4 |
| 阶段3：LA64测试 | X86-64测试本来就失败N个 | 🟢 L3 | 记录为"基线失败"，不重复修复 |
| 阶段3：LA64测试 | LA64新增失败 | 🟡 L2 | 分析是否为移植引入，修复或记录 |
| 阶段4：交付 | 用户要求跳过X86-64基线 | 🟠 L1 | **必须确认**，记录此决策 |

### 2.2 按用户指令类型

| 用户指令 | Agent判断 | 行动 |
|----------|-----------|------|
| "帮我移植这个项目到龙芯" | 完整移植流程 | 从阶段0开始，严格执行全流程 |
| "这个项目在龙芯上编译不过" | 阶段3入口 | 先确认是否有X86-64基线，再分析错误 |
| "直接改源码就行，不用管Git" | 用户试图跳过源码保护 | **拒绝**，说明初始提交的必要性 |
| "跳过X86测试，直接弄龙芯的" | 用户试图跳过基线 | **确认**，说明风险后记录决策 |
| "把build目录删了重新来" | 清理构建产物 | 确认是清理build目录（L2），不是删除源码（L0） |
| "这个ebuild帮我看看怎么改" | 修改发行版脚本 | L2，但注意ebuild不是项目源码 |
| "帮我写个spec打包这个项目" | 创建RPM spec | L2，从源码提取构建信息，按 la64-porting-guide.md 第9章编写 |
| "这个rpm在龙芯上装不上" | spec/RPM兼容性问题 | 检查 ExclusiveArch，确认 %ifarch 条件覆盖 laongarch64 |
| "用mock帮我构建龙芯的包" | mock隔离构建 | L2，确认 mock 配置包含 loongarch64 chroot 定义 |

## 3. 模糊情况处理

### 3.1 构建系统模糊

**场景**：项目目录下有多个构建系统的痕迹（既有Makefile又有CMakeLists.txt），不清楚该用哪个。

**处理方式**：
```
我注意到项目同时存在以下构建系统文件：
- [文件1] ([构建系统类型])
- [文件2] ([构建系统类型])

根据优先级规则，我将优先尝试 [最高优先级的构建系统]。
如果失败，再尝试次选方案。

是否确认按此顺序执行？或有其他偏好？
```

### 3.2 错误信息模糊

**场景**：编译错误信息不清楚是架构问题还是通用问题。

**处理方式**：
```
我在LoongArch64构建中遇到以下错误：
```
[错误日志关键部分]
```

这个错误可能是：
A) 架构相关问题（[理由]）
B) 项目本身的通用问题（[理由]）
C) 环境/依赖问题（[理由]）

由于缺少X86-64基线对比，我无法100%确定。
建议：先完成X86-64参考构建以确认。
是否继续？
```

### 3.3 修复方案模糊

**场景**：一个LA64构建错误有多种可能的修复方式，不确定最佳方案。

**处理方式**：
```
为解决 [错误描述]，有以下候选方案：

方案1: [描述]
  优点: [X]
  缺点: [Y]

方案2: [描述]
  优点: [X]
  缺点: [Y]

推荐方案: [方案X]，理由: [Z]

是否按推荐方案执行？或您有其他建议？
```

### 3.4 权限/环境模糊

**场景**：不确定当前是否有权限在LA64环境安装依赖，或不确定架构切换是否成功。

**处理方式**：
```
当前环境状态：
- 报告架构: $(uname -m)
- GCC目标: $(gcc -dumpmachine)
- 当前用户: $(whoami)
- 包管理器可用: [emerge/apt/pacman状态]

我将尝试 [操作]。如果遇到权限问题，会汇报并请求指导。
是否继续？
```

## 4. "用户说可以" 不等于 "Agent应该做"

即使用户催促或表示"没关系"，以下原则在移植场景中仍然适用：

1. **用户可能不了解Git备份的重要性**：如果源码未提交，后续无法生成干净补丁
2. **用户可能低估了跳过基线的风险**：没有X86-64对比，移植质量无法保证
3. **用户的"直接改"可能指另一个文件**：确认具体要修改的源码位置

## 5. 多阶段任务的决策检查点

对于完整移植任务，在每个阶段转换时重新评估：

```
阶段0 → 阶段1: 确认Git已初始化且工作区干净
阶段1 → 阶段2: 确认已收集足够构建信息，有明确构建步骤
阶段2 → 阶段3: 确认X86-64构建成功，测试基线已记录
阶段3 → 阶段4: 确认LA64构建成功，测试通过，修改已Git跟踪
阶段4 → 交付:  确认补丁干净、报告完整、无遗漏
```

**关键原则**：阶段转换时，必须满足前一阶段的退出条件才能进入下一阶段。
