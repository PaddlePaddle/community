# 安全操作规范

> **⚠️ 最高优先级文档**
> 
> 本文档规定Agent在OpenClaw移植项目中允许和禁止的操作。
> 
> **红色区域的操作如果未经明确授权执行，可能导致源码丢失、构建污染或移植失败。**

## 1. 操作分级系统

所有操作按风险等级分为四类：

| 等级 | 颜色 | 含义 | Agent行为 |
|------|------|------|-----------|
| L0 - 禁止 | 🔴 | 绝对禁止，无例外 | **立即停止并拒绝** |
| L1 - 需授权 | 🟠 | 必须获得用户显式确认 | **执行前必须询问** |
| L2 - 需审查 | 🟡 | 可以执行但需谨慎，建议记录 | **执行后汇报** |
| L3 - 常规 | 🟢 | 安全操作，自由执行 | **直接执行** |

---

## 2. L0 - 禁止操作（🔴）

以下操作**在任何情况下都不允许执行**：

### 2.1 源码与版本控制

- [ ] **在未执行`git init && git add -A && git commit -m "Initial import"`的情况下修改源码**
- [ ] 删除 `.git` 目录或其中的任何内容
- [ ] 执行 `git reset --hard` 或 `git push -f`（可能丢失移植进度）
- [ ] 直接覆盖原始源码文件而不通过Git跟踪（如用 `cp` 或重定向覆盖）
- [ ] 将构建产物（.o, .so, 可执行文件）提交到Git仓库

### 2.2 构建与编译

- [ ] 在LoongArch64环境编译X86-64二进制（除非明确使用交叉编译工具链）
- [ ] 混用X86-64和LoongArch64的构建产物（不清除build目录就切换架构构建）
- [ ] 执行 `make clean` 或删除build目录前，未确认其中有价值的内容
- [ ] 修改系统级编译器/工具链配置（如 `/usr/bin/gcc` 的符号链接）

### 2.3 系统与环境

- [ ] `rm -rf /` 或任何递归删除根目录的操作
- [ ] `chmod 777` 源码目录（过度放宽权限可能导致安全问题）
- [ ] 修改 `/etc/` 下的系统配置文件
- [ ] 在不确定架构的情况下执行编译（必须先 `uname -m` 确认）

### 2.4 网络与外部资源

- [ ] 从不可信来源下载补丁或替换文件
- [ ] 自动提交代码到上游仓库（必须人工审核）

---

## 3. L1 - 需授权操作（🟠）

以下操作**必须**在执行前获得用户的明确确认：

### 3.1 源码修改

- [ ] 修改项目的核心源码（非构建配置）以适配LoongArch64
- [ ] 添加/修改内联汇编代码（`__asm__` 或 `asm()`）
- [ ] 修改与字节序、对齐、指针宽度相关的代码
- [ ] 删除项目中看似"无用"的文件（可能是平台特定文件）

### 3.2 构建系统修改

- [ ] 修改 `CMakeLists.txt`、`configure.ac`、`Makefile.am` 等构建系统文件
- [ ] 添加新的编译条件或宏定义（如 `__loongarch__`）
- [ ] 修改 `*.ebuild`、`PKGBUILD` 或 `*.spec`（这些是发行版的打包脚本，非项目源码）
- [ ] 升级/降级项目依赖版本

### 3.3 架构切换相关

- [ ] 通过Node/SSH方式切换运行时架构环境
- [ ] 在容器中运行可能影响宿主机的操作
- [ ] 使用 `setarch` 或类似命令改变报告的架构

### 3.4 测试与验证

- [ ] 跳过X86-64参考构建直接进入LoongArch64移植
- [ ] 跳过测试直接标记移植完成

**确认模板**（Agent必须使用类似表述）：

```
我即将执行 [具体操作]，这属于 L1 风险操作。

操作详情：
- 目标文件：[文件路径]
- 当前架构：[x86_64/loongarch64]
- 潜在影响：[描述可能影响的功能或兼容性]

请确认是否继续？（请回复"确认执行"或"取消"）
```

---

## 4. L2 - 需审查操作（🟡）

以下操作可以执行，但Agent应当：

1. 在执行前简要说明计划
2. 执行后汇报结果
3. 保留操作记录（如适用）

- [ ] 创建新的构建目录（`mkdir build`）
- [ ] 运行构建命令（`make`, `cmake --build`, `ninja` 等）
- [ ] 运行测试命令（`make test`, `ctest` 等）
- [ ] 修改 `.gitignore` 文件
- [ ] 在源码中添加调试输出（printf/logging）
- [ ] 安装构建依赖（`emerge`, `pacman`, `apt` 等包管理器）
- [ ] 清理构建产物（`make clean`, `rm -rf build/`）

---

## 5. L3 - 常规操作（🟢）

以下操作可以自由执行，无需额外确认：

- [ ] 读取代码、配置文件、构建文档、ebuild、PKGBUILD、spec
- [ ] 执行 `git status`, `git log`, `git diff` 等只读Git操作
- [ ] 检查系统状态（`uname -m`, `gcc -dumpmachine`, `ps`, `df` 等）
- [ ] 在开发环境启动/停止服务
- [ ] 修改Agent自身文档（`src/agent-docs/` 目录）
- [ ] 创建移植记录/分析笔记

---

## 6. 架构环境识别指南

Agent必须能够识别当前架构，以应用正确的安全等级：

### 6.1 架构判断命令

```bash
# 必做：确认CPU架构
uname -m
# 输出: x86_64      → 当前在X86-64环境
# 输出: loongarch64 → 当前在LoongArch64环境
# 输出: 其他        → 未知环境，暂停并询问

# 必做：确认GCC目标平台
gcc -dumpmachine
# 输出: x86_64-linux-gnu       → X86-64工具链
# 输出: loongarch64-linux-gnu  → LoongArch64工具链
# 输出: 其他                    → 可能是交叉编译环境
```

### 6.2 构建产物隔离检查

切换架构后，构建前必须检查：

```bash
# 如果存在旧构建目录，确认其架构
file build/src/some_binary 2>/dev/null || echo "No existing binary"
# 输出应与新架构匹配，如果不匹配，必须清理后重新构建

# 推荐的隔离做法
rm -rf build/          # 清除旧构建产物
mkdir build && cd build
# 然后执行新的配置和构建
```

### 6.3 默认假设

**如果无法明确判断架构，默认按可能影响源码完整性的最高谨慎级别处理。**

## 7. 源码保护检查流程

每次Agent开始处理新项目时，必须执行以下检查：

```bash
# Step 1: 检查Git是否已初始化
if [ ! -d .git ]; then
    echo "Git未初始化，执行初始提交..."
    git init
    git add -A
    git commit -m "Initial import for LoongArch64 porting"
    echo "初始提交完成，commit: $(git rev-parse HEAD)"
else
    echo "Git已初始化"
    git status
fi

# Step 2: 确认工作区干净（或已提交）
if [ -n "$(git status --porcelain)" ]; then
    echo "警告：工作区有未提交更改"
    git status --short
    # 询问用户是否先提交当前更改
fi

# Step 3: 记录当前状态
echo "=== 项目初始状态 ==="
echo "目录: $(pwd)"
echo "Git HEAD: $(git rev-parse HEAD)"
echo "架构: $(uname -m)"
echo "GCC: $(gcc -dumpmachine 2>/dev/null || echo 'N/A')"
```

## 8. 紧急情况处理

如果Agent发现以下情况，应立即停止并告警：

1. **源码未备份但用户要求直接修改** → 拒绝执行，要求先 `git init`
2. **在LoongArch64上发现X86-64二进制未清理** → 可能导致链接错误，需清理build目录
3. **用户要求执行L0级别操作**（如 `git reset --hard`） → 拒绝并说明风险
4. **发现项目源码似乎已被修改且未提交** → 暂停，要求用户确认状态

**告警模板**：

```
🚨 安全告警
检测到：[问题描述]
风险等级：[L0/L1/L2]
涉及文件/操作：[具体描述]

建议行动：[停止操作/先提交/清理build目录]
我不会继续执行相关操作，请人工确认后指示下一步。
```
