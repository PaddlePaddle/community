- [ ] 我已确认当前所处架构（X86-64 或 LoongArch64）
- [ ] 源码已完成"初始提交"或备份（首次操作时）

## Agent 行为契约

### 1. 架构感知原则

- **每次执行命令前，确认当前架构**：`uname -m` 或 `gcc -dumpmachine`
- **不要假设当前环境就是目标架构**。Agent可能在X86-64上运行，也可能通过SSH/Node切换到LoongArch64
- **构建产物具有架构特异性**。X86-64上编译的结果不能在LoongArch64上运行，反之亦然

### 2. 源码保护原则

- **首次接触项目必须进行"初始提交"**：`git init && git add -A && git commit -m "Initial import"`
- **所有移植修改必须在Git跟踪下进行**，禁止直接覆盖原始文件而不留痕迹
- **移植补丁必须与原始源码可区分**，便于回溯和提交上游

### 3. 基线先行原则

- **任何移植工作开始前，先在X86-64上完成参考构建**
- **没有X86-64基线，不开始LoongArch64移植**
- 如果项目在X86-64上都构建失败，应先解决通用构建问题

### 4. 变更最小化原则

- 移植修改应**仅限于解决架构差异**，不做功能性改动
- 不要"顺便"升级依赖、重构代码或修改无关配置
- 优先使用条件编译（`#ifdef __loongarch__`）而非硬编码修改

### 5. 显式确认原则

以下情况必须停止并向用户确认：

- 修改可能影响X86-64兼容性的代码
- 删除原始源码文件（即使是构建生成的，也应先确认）
- 在源码树中引入新的二进制文件或预编译对象
- 用户要求跳过X86-64参考构建直接进入移植
- 发现项目没有构建文档且构建系统不明

### 6. 文档同步原则

- 发现新的构建依赖或步骤，更新到移植记录中
- 解决移植问题后，在 `common-pitfalls.md` 中记录方案
